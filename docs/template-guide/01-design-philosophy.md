# 01 — 设计思路与哲学

## 核心理念

> **把专业设计师的决策过程编码成数据，让 AI 编码助手也能做出专业级设计决策。**

---

## 为什么需要这个项目？

### AI 编码助手的设计盲区

AI 编码助手（Claude、Cursor、Copilot 等）擅长写代码，但在 UI/UX 设计决策上存在系统性弱点：

| 问题 | 表现 | 根因 |
|------|------|------|
| 配色随机 | 每次对话生成不同颜色 | 缺乏行业配色知识 |
| 风格不一致 | 同一项目混用多种风格 | 没有风格约束系统 |
| 字体搭配差 | 随机选择字体 | 不了解字体搭配理论 |
| 忽略最佳实践 | 触摸目标太小、对比度不够 | 缺乏 UX 规则库 |
| 无行业针对性 | 银行 App 用了电商的设计 | 不理解行业设计规范 |

### 解决方案：用数据驱动设计

与其教 AI "什么是好设计"，不如直接给它一个**专业设计知识库**：

```
传统方式:  用户说 "做一个 SaaS 仪表盘" → AI 凭感觉设计
本项目:    用户说 "做一个 SaaS 仪表盘" → 搜索 161 种产品知识库
           → 匹配 "SaaS (General)" → 推荐 Glassmorphism + 信任蓝配色
           → AI 按照专业推荐执行
```

---

## 五大设计原则

### 1. 数据驱动 > 规则编码

知识存储在 CSV 中，而非硬编码在代码里。

**优点**:
- 非开发者也能维护（Excel 编辑 CSV）
- 新增行业只需加一行数据
- 不同版本的知识可以 diff 对比

### 2. 搜索 > 分类

不用树形分类（`行业 > 子行业 > 风格`），而是用搜索。

**原因**:
- 用户的描述千变万化，无法穷举分类路径
- BM25 搜索天然支持模糊匹配和关键词组合
- 搜索结果按相关性排序，比固定分类更灵活

### 3. 推理引导 > 直接返回

不直接返回"你应该用蓝色"，而是通过推理链：

```
产品类型 → 推理规则 → 影响搜索 → 最终推荐
```

**原因**:
- 设计决策有上下文依赖
- 同一产品类型在不同场景下可能需要不同风格
- 推理链让 AI 能理解"为什么"，从而做出更好的判断

### 4. 平台无关 > 锁定生态

同一份 Skill 内容适配 18 个 AI 平台。

**实现**:
- 内容与配置分离
- 占位符系统 + 平台 JSON
- 添加新平台零代码修改

### 5. 渐进增强 > 全有或全无

用户可以只用搜索（1 条命令），也可以用完整设计系统生成（4 步流程）。

## 架构哲学

### 三层分离

```
┌─────────────────────────┐
│  表现层 (Skill File)      │  ← AI 读取的 Markdown
├─────────────────────────┤
│  逻辑层 (Python Scripts)  │  ← 搜索引擎 + 推理引擎
├─────────────────────────┤
│  数据层 (CSV Files)       │  ← 可独立维护的知识库
└─────────────────────────┘
```

每一层可以独立演进：
- 改数据不动代码
- 改代码不动表现
- 改表现不动逻辑

### 配置驱动

一切差异通过配置表达，而非 if-else：

```python
# 不是这样:
if platform == "claude":
    include_quick_ref = True
elif platform == "cursor":
    include_quick_ref = False

# 而是这样:
config = load_platform_config(platform)
include_quick_ref = config["sections"]["quickReference"]
```

## 为什么选择这些技术？

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|---------|------|
| 数据格式 | CSV | JSON/SQLite/YAML | Excel 兼容、Git diff 友好、零依赖 |
| 搜索算法 | BM25 | TF-IDF/Embedding | 轻量、无外部依赖、效果够好 |
| 脚本语言 | Python | Node.js/Bash | 几乎所有开发者都有 Python |
| CLI 语言 | TypeScript | Python/Bash | npm 生态更适合 CLI 分发 |
| 包管理 | npm | pip/homebrew | 全球覆盖最广的包管理器 |
