# 01 — 执行摘要：AI Skill 通用框架

## 一句话定义

> **AI Skill 通用框架** = 一套将任何领域的专业知识打包成 AI 编码助手可消费的结构化知识库的架构模式。

---

## 框架解决的问题

### AI 编码助手的知识盲区

当前的 AI 编码助手（Claude Code、Cursor、GitHub Copilot 等）擅长写代码，但在**垂直领域的专业决策**上存在系统性弱点：

| 问题 | 表现 | 根因 |
|------|------|------|
| 知识不一致 | 每次对话给出不同建议 | 缺乏固定知识库 |
| 缺乏行业针对性 | 银行 App 用了游戏设计 | 不理解行业规范 |
| 无法遵循最佳实践 | 忽略可访问性、性能 | 缺乏规则库 |
| 推荐不可解释 | "选蓝色"但说不出为什么 | 无推理链 |
| 平台碎片化 | Claude 能用的 Cursor 用不了 | 每个平台格式不同 |

### 框架的解决方案

```
传统方式:
  用户请求 → AI 凭内部知识生成 → 质量不稳定

框架方式:
  用户请求 → AI 读取 Skill → 搜索专业知识库 → 推理引擎匹配规则 → 基于专业数据生成 → 质量稳定
```

---

## 核心价值主张

### 1. 知识外部化 (Externalized Knowledge)

将专业知识从 AI 的内部权重中提取出来，存储到**可审计、可编辑、可版本控制**的外部数据文件中。

```
AI 内部知识:  不透明、不稳定、不可控
→
外部知识库:  CSV 文件、可 Git diff、可人工审核、可团队协作
```

### 2. 搜索驱动 (Search-Driven)

不用固定分类树（会过时且僵化），而是用 BM25 全文搜索，天然支持模糊匹配、同义词、组合查询。

```
传统分类:  行业 > 子行业 > 功能 > 子功能 → 路径僵化
搜索驱动:  "fintech mobile dark mode" → BM25 排序 → Top 3 结果
```

### 3. 推理增强 (Reasoning-Enhanced)

不只是返回搜索结果，还通过推理引擎建立「产品类型 → 决策规则 → 搜索引导」的完整推理链。

```
搜索结果:  "这里有 3 种颜色方案"
推理增强:  "你做的是金融产品 → 应该优先信任蓝 → 在 3 种中选了信任蓝 → 因为 Decision_Rules 说 if_financial: use-trust-blue"
```

### 4. 一次开发，N 个平台 (Write Once, Run Anywhere)

一份知识库内容，通过模板系统 + 平台配置 JSON，自动适配任意数量的 AI 平台。

```
1 份 skill-content.md 模板
+ N 份 platform.json 配置
= N 个平台的 Skill 文件
```

### 5. 零外部依赖 (Zero External Dependencies)

Python 搜索引擎只依赖标准库（csv、json、math、re、os），确保在任何环境都能运行。

---

## 框架适用场景

### ✅ 适合

| 场景 | 示例 |
|------|------|
| AI 编码助手的领域增强 | UI/UX 设计、API 设计、数据库建模 |
| 团队编码规范标准化 | 代码风格、命名规范、测试规范 |
| 行业知识注入 | 医疗法规、金融合规、法律条文 |
| 技术栈最佳实践 | React、Flutter、Spring 指南 |
| 多平台内容分发 | 同一知识库给 18 个 AI 平台使用 |

### ❌ 不适合

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| 实时数据查询 | CSV 是静态快照 | 接 API / 数据库 |
| 超大数据集 (>100MB) | BM25 内存搜索有上限 | SQLite / Elasticsearch |
| 需要向量语义搜索 | BM25 是关键词匹配 | 加 Embedding 层 |
| 非文本知识 | CSV 只存文本 | 多模态存储 |

```
│  └─────────────────────────────────────────────────────┘    │

│  特性: 零依赖 | 18 平台 | 搜索驱动 | 推理增强 | 可扩展    │
└─────────────────────────────────────────────────────────────┘
```

## 关键数字（参考实现）

| 指标 | 值 | 说明 |
|------|-----|------|
| 数据文件 | 30 CSV | 14 主域 + 16 技术栈 |
| 总数据行 | 7,258 | 跨所有 CSV |
| 支持平台 | 18 | AI 编码助手平台 |
| 搜索延迟 | ~50ms | 单域 BM25 查询 |
| 生成延迟 | ~525ms | 完整设计系统 5 域聚合 |
| Python 依赖 | 0 | 纯标准库 |
| CLI 依赖 | 4 | commander、chalk、ora、prompts |
| 包体积 | ~564KB | npm 包含全部资源 |

## 技术栈选型

| 组件 | 技术选择 | 替代方案 | 选择理由 |
|------|---------|---------|---------|
| 数据存储 | CSV | JSON / SQLite / YAML | Excel 兼容、Git diff 友好、零解析依赖 |
| 搜索算法 | BM25 | TF-IDF / Embedding / Full-text | 零外部依赖、可预测、效果够好 |
| 脚本语言 | Python | Node.js / Bash | 覆盖率最广、标准库够用 |
| CLI 语言 | TypeScript | Python / Go | npm 生态最适合 CLI 分发 |
| 构建工具 | Bun | tsc / esbuild / webpack | 快速、单文件输出、TS 原生 |
| 包管理 | npm | pip / homebrew | 全球覆盖最广的包管理器 |
| 模板语法 | `{{PLACEHOLDER}}` | Handlebars / EJS / Jinja2 | 极简、无依赖、易理解 |
