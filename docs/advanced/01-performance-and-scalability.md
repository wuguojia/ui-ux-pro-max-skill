# 01 — 性能与可扩展性分析

## 搜索性能

### BM25 时间复杂度

| 操作 | 时间复杂度 | 实际耗时 |
|------|-----------|---------|
| fit (建索引) | O(N * L) | ~50ms (161 文档) |
| score (查询) | O(Q * D) | ~5ms |
| 完整搜索 (加载+索引+查询) | - | ~100ms |

N=文档数, L=平均文档长度, Q=查询词数, D=倒排索引匹配文档数

### 瓶颈分析

```
python3 search.py "SaaS dashboard" --domain style

时间分布:
  Python 启动:     ~30ms  (不可避免)
  CSV 加载:        ~20ms  (IO 瓶颈)
  BM25 fit:        ~15ms  (CPU)
  BM25 score:      ~5ms   (CPU)
  输出格式化:       ~2ms
  总计:            ~72ms
```

**核心发现**: Python 启动和 CSV 加载占 70%+ 时间。搜索算法本身很快。

### 设计系统生成性能

```
python3 search.py "fintech" --design-system

时间分布:
  产品搜索:        ~80ms  (加载 products.csv + 搜索)
  推理加载:        ~30ms  (加载 ui-reasoning.csv)
  多域搜索 x5:     ~400ms (style + color + landing + typography)
  最佳匹配选择:    ~5ms
  格式化输出:       ~10ms
  总计:            ~525ms
```

## 数据可扩展性

### 当前规模

| 指标 | 数值 |
|------|------|
| CSV 文件数 | 30 |
| 总行数 | 7,318 |
| 最大文件 | google-fonts.csv (728KB) |
| 总数据量 | ~1.2MB |

### 扩展预测

| 数据量 | BM25 性能 | 加载时间 | 可行性 |
|--------|----------|---------|--------|
| 1K 行 | <10ms | <50ms | 完全无压力 |
| 10K 行 | <50ms | <200ms | 良好 |
| 100K 行 | <500ms | <2s | 需要缓存 |
| 1M 行 | ~5s | ~20s | 需要索引 |

**结论**: 当前架构支持 10 万行以内的数据无需任何优化。

### 如果需要更大规模

1. **SQLite**: 替换 CSV，支持索引查询
2. **预计算索引**: 启动时 fit() 一次，缓存到文件
3. **增量加载**: 只加载请求的域，不加载全部 CSV
4. **TypeScript 重写**: core.ts 已存在，Node.js 启动更快

## 平台可扩展性

### 添加新平台的成本

| 步骤 | 工作量 | 耗时 |
|------|--------|------|
| 创建 platform JSON | 复制+改 5 行 | 1 分钟 |
| 注册 AIType | 加 1 个字符串 | 30 秒 |
| 添加检测规则 | 加 1 行映射 | 30 秒 |
| **总计** | | **~2 分钟** |

### 域可扩展性

添加新搜索域：
| 步骤 | 工作量 |
|------|--------|
| 创建 CSV | 准备数据（主要工作） |
| 注册 CSV_CONFIGS | 加 5 行配置 |
| 添加检测关键词 | 加 1 行正则 |
| 更新 Skill 模板 | 加 1 行搜索参考 |

## TypeScript 重写的性能优势

core.ts (680 行) 已存在，是 core.py 的 1:1 重写。

| 指标 | Python | TypeScript (Node) |
|------|--------|-------------------|
| 启动时间 | ~30ms | ~10ms |
| CSV 解析 | csv 模块 | 自定义 RFC-4180 |
| 外部依赖 | 0 | 0 |
| 类型安全 | 无 | 完整接口定义 |

TypeScript 版本在 CLI 场景下更有优势（Node.js 已经在运行）。
